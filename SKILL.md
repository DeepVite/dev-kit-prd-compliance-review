# 工程审查与需求对齐审查

## 核心目标

> 当前开发结果是否符合 PRD？
> 如果不符合，哪些必须先修，哪些需要产品经理确认？

这不是代码审查，是**工程审查**——只检查产品符合性，不检查代码风格。

## 触发场景

- 开发完成后、提交 PR 前
- 合并前自检

## 审查流程

```text
1. 拆 PRD → 把 PRD 拆成可验证条目
2. 自检   → 开发者先自己跑一遍
3. 对照   → 逐条对照 PRD 检查产品符合性
4. 标记   → 标记偏离类型和严重等级
5. 修复   → 先修复 S0/S1
6. 整理   → 把需要产品决策的问题整理给产品经理
```

### 第一步：拆 PRD

把 PRD 拆成可检查的条目，每条至少包含：

| 字段 | 说明 |
|---|---|
| PRD 编号 | FR-001、BR-001、UI-001、ANA-001 等 |
| PRD 要求 | 产品要求是什么 |
| 是否必须 | 必须 / 可选 |
| 验证方式 | 怎么证明它实现了（手动测试 / 截图 / 录屏 / 日志 / 埋点） |

### 第二步：自检清单

- [ ] 主流程是否跑通？
- [ ] 必须实现的 PRD 是否都实现？
- [ ] 业务规则是否正确（金额、权限、状态、次数、时间）？
- [ ] 异常情况是否处理（空状态、失败、无权限、重复点击）？
- [ ] 页面、文案、交互是否符合 PRD？
- [ ] 数据结果是否正确？
- [ ] 埋点是否符合 PRD？
- [ ] 是否有截图、录屏、日志或测试结果作为证据？

### 第三步：产品符合性检查

只看以下维度，不检查代码风格：

| 检查项 | 核心问题 |
|---|---|
| 功能 | PRD 要求的功能是否实现？ |
| 流程 | 用户操作流程是否符合 PRD？ |
| 业务规则 | 金额、权限、状态、次数、时间等规则是否正确？ |
| 页面文案 | 文案、按钮、提示是否符合 PRD？ |
| 异常状态 | 空状态、失败状态、无权限、重复点击是否处理？ |
| 数据/埋点 | 数据结果、埋点是否符合 PRD？ |

### 第四步：标记偏离

所有和 PRD 不一致的地方都要标记：

| 偏离类型 | 说明 |
|---|---|
| 无偏离 | 实现和 PRD 一致 |
| 正偏离 | 超出 PRD，但可能更好 |
| 负偏离 | 没达到 PRD 或和 PRD 冲突 |
| PRD 不明确 | PRD 没写清楚，无法判断 |

### 第五步：标记严重等级

| 等级 | 名称 | 说明 | 处理方式 |
|---|---|---|---|
| S0 | 阻断级 | 不能上线，存在重大业务、安全、数据风险 | 必须修复 |
| S1 | 严重级 | 核心需求不符合或主要流程受影响 | 修复后再合并 |
| S2 | 一般级 | 非核心问题，有明确替代方案 | 可沟通后决定 |
| S3 | 轻微级 | 文案、样式等轻微问题 | 可后续优化 |
| S4 | 建议级 | 更好的实现方式或优化建议 | 不阻塞上线 |

### 第六步：处理规则

| 问题等级 | 程序员处理方式 |
|---|---|
| S0 | 自己先修，不能交给产品验收 |
| S1 | 自己先修，修完再提交 |
| S2 | 整理影响和方案，找产品确认 |
| S3 | 可记录，必要时找产品确认 |
| S4 | 记录为建议，不阻塞 |

**核心规则：**

> S0/S1 不应该直接丢给产品经理。
> 产品经理主要看 S2/S3、正偏离、PRD 不明确的问题。

## 输出

审查完成后，按 `reference\report-template.md ` 的格式输出工程审查报告。
**报告切勿过度复杂！报告应该精简高效！**

报告分两种用途：
1. **给程序员自己看**——先发现并解决严重问题
2. **给产品经理看**——整理需要产品决策的问题

## 需要读取的资料

- 输出报告前，读取 `reference\report-template.md ` 获取完整报告模板。
